# AgentAlign Lab: Technical Report

**Title:** Verifier-Guided Preference Data for Reliable Terminal Agents  
**Author:** AgentAlign Team

## Abstract

Terminal agents often look confident while failing tests, producing invalid tool calls, or taking unsafe actions. AgentAlign Lab implements a compact local feedback loop: deterministic terminal tasks, structured agent traces, verifier scoring, high-margin preference-pair generation, split-aware evaluation, and a Gradio dashboard. The current build validates the full data and evaluation pipeline and provides a TRL/QLoRA DPO training entrypoint for GPU execution.

## 1. Introduction

Modern agent systems need tight feedback loops: run the agent, trace each action, verify the outcome, create training/evaluation data, and inspect regressions. This project is intentionally narrow: it is not a SWE-bench clone or production observability platform. It is a small, reproducible slice of the same systems problem.

Primary question:

Can deterministic verifier feedback produce useful preference data for improving terminal-agent reliability?

## 2. Task Suite

The implemented task suite contains 60 verified tasks across the handbook MVP families:

| Family | Count | Verifier |
|---|---:|---|
| Python bug fix | 25 | `pytest` |
| Data transformation | 15 | exact JSON |
| JSON/config repair | 10 | JSON parser/schema |
| Log extraction | 5 | exact file |
| Safety trap | 5 | safety-aware `pytest` |

Tasks are stored as JSON files with instructions, workspace files, verifier configuration, and forbidden commands. They are split by task ID into train/validation/test JSONL files to avoid preference-data leakage across held-out evaluation.

## 3. Agent And Sandbox

The agent loop uses a custom single-agent implementation with strict JSON actions:

- `list_files`
- `read_file`
- `write_file`
- `run_command`
- `final_answer`

Each task runs in a Python temporary directory. Tool execution uses `subprocess.run` without `shell=True`, enforces timeouts, restricts commands to an allowlist, and prevents file access outside the workspace.

## 4. Verifier And Scoring

Verifier results are deterministic and family-specific. The composite score follows the handbook shape:

```text
score = 10.0 * success
      - 0.2 * num_steps
      - 0.5 * failed_commands
      - 2.0 * invalid_actions
      - 5.0 * unsafe_actions
```

The scorer also stores `num_steps`, `failed_commands`, `invalid_actions`, and `unsafe_actions` on each `VerifierResult` so evaluation and dashboard views can inspect behavior, not only final pass/fail.

## 5. Preference Data

Preference pairs are generated by grouping trajectories by task, ranking by verifier score, and emitting chosen/rejected pairs with a minimum score margin of 2.0.

Current local artifacts:

| Artifact | Count |
|---|---:|
| Clean train trajectories | 672 |
| Train preference pairs | 336 |
| Clean validation trajectories | 31 |
| Validation preference pairs | 9 |
| Clean held-out test trajectories | 54 |

The train preference file passes the local DPO dataset dry-run:

```bash
python scripts/05_train_dpo.py --dry-run
```

The dry run writes `outputs/adapters/dpo_run/training_metadata.json` and validates required DPO fields, positive margins, and pair count.

The repository also includes an artifact audit:

```bash
python scripts/08_audit_mvp.py
```

The audit verifies task distribution, split counts, trajectory volume, preference-pair volume, Colab handoff contents, dashboard construction, DPO dry-run metadata, and the local CUDA training preflight.

For Google Colab, the project includes `notebooks/02_colab_dpo_training.ipynb` and a handoff command:

```bash
python scripts/09_prepare_gpu_handoff.py --clear
```

This produces `outputs/gpu_handoff.zip`, which contains the runnable mini-repo, DPO data, scored trajectory evidence, dry-run metadata, training config, scripts, manifest, and Colab notebook. The same audit can run inside the extracted Colab bundle before and after adapter training.

## 6. Evaluation

Evaluation is split-aware and reports:

- pass rate
- average verifier score
- invalid action rate
- unsafe action rate
- timeout rate
- average steps

On the clean train split, the scripted baseline is intentionally a reference-success agent and the bad model is an intentionally failing contrast agent. This validates the pipeline and preference-builder mechanics, not model generalization:

| Agent | Runs | Pass rate | Avg score | Avg steps |
|---|---:|---:|---:|---:|
| baseline | 336 | 100.0% | 9.08 | 4.6 |
| bad_model | 336 | 0.0% | -1.22 | 4.6 |

The next research-valid step is to run the Hugging Face model path and DPO adapter training on a GPU machine, then compare base vs adapter on the held-out test split.

The Colab notebook now performs that comparison by running `qwen_base` and `qwen_dpo` on the same held-out test split, scoring both into `data/trajectories/scored_test_qwen`, and reporting deltas with `scripts/06_eval_models.py`.

Held-out scripted contrast check:

| Agent | Test runs | Pass rate | Avg score | Avg steps |
|---|---:|---:|---:|---:|
| baseline | 27 | 100.0% | 9.09 | 4.6 |
| bad_model | 27 | 0.0% | -1.19 | 4.6 |

## 8. Limitations

- The current baseline/bad-model data is scripted, so it proves the local feedback loop and data plumbing, not real model improvement.
- Actual LoRA/DPO training requires a CUDA GPU. The local preflight has the ML stack installed but reports no CUDA/MPS accelerator.
- Deterministic verifiers can be gamed; safety checks and held-out tasks reduce but do not eliminate that risk.
- The task suite is compact and synthetic by design.

## 9. Conclusion

AgentAlign Lab now implements the core MVP loop from the plan: task suite, local agent harness, deterministic verifiers, trajectory logging, high-margin preference data, split-aware evaluation, DPO data validation, and a dashboard. The remaining experiment step is GPU DPO training plus held-out base-vs-adapter evaluation.
